Here is the complete, detailed MVP specification for the **Adaptive Heartbeat & Memory Governance System** — now fully embedded as a core subsystem of the Tesseract Hyperkernel.

---

## 1. The Core Idea: The Hyperkernel as the "Sovereign" System

The Tesseract Hyperkernel is no longer just a hypervisor. It is a **sovereign operating system** for other kernels. It does not delegate its authority to a separate "Guardian" process. Instead, it integrates the heartbeat, memory, and execution logic directly into its main scheduling loop. 

This means:
- **There is no separate process to compromise.** The Hyperkernel *is* the watcher.
- **There is no inter-process communication overhead.** All decisions are made synchronously within the Hyperkernel's critical path.
- **The logic is simpler and more auditable.** It is a single, centralized state machine.

The Hyperkernel enforces a **"Dynamic Resource Constitution"** – a set of rules that every user kernel must follow, or face immediate, automated consequences.

---

## 2. Detailed Implementation Strategy

### Component A: The Constitution (The Rules of Existence)

Before a kernel is even spawned, it signs a "constitution" with the Hyperkernel. This defines the rules of engagement.

- **The Contract:** When the Hyperkernel calls `kspawn()`, it loads the kernel image and assigns it:
    - A unique Kernel ID.
    - A base memory allocation (e.g., 16 MB).
    - A priority class: `Critical`, `High`, `Normal`, or `Idle`.
- **The Obligations:** The kernel implicitly agrees to:
    - Respond to every heartbeat request.
    - Provide accurate memory usage and requests.
    - Make at least one syscall per second to prove it is not hung.

### Component B: The Dynamic Heartbeat Subsystem (Integrated into Scheduler)

The heartbeat is no longer a separate interrupt or timer. It is integrated into the Hyperkernel's scheduler tick.

- **The Tick:** The Hyperkernel's main loop runs at a variable frequency. It calculates the optimal interval based on the total system load.
- **The Ping Phase:** At the beginning of each tick, the Hyperkernel broadcasts a "heartbeat check" to all active kernel instances via a hardware-interrupt style signal (a logical signal, not a separate process).
- **The Response Window:** The Hyperkernel waits for exactly **2x the current tick interval** (dynamic) for a response from each kernel.
- **The Payload:** The response is a lightweight packet containing:
    - The kernel's current memory footprint.
    - A memory request for the next cycle.
    - A "health score" (generated by the kernel's own internal driver checks).
    - A timestamp of the last successful syscall.

### Component C: The Memory Governance Subsystem (Integrated into Allocation)

Memory is allocated and revoked in the same tick cycle as the heartbeat, ensuring a strict, predictable budget.

- **The Request Cycle:** During its heartbeat response, each kernel must explicitly request the amount of physical memory it requires for the *next* tick.
- **The Aggregation Phase:** The Hyperkernel sums all `memory_requested` values.
- **The Arbitration Phase:**
    - **If Total Requested ≤ Total Memory:** The Hyperkernel grants every kernel exactly what it requested.
    - **If Total Requested > Total Memory (Congestion):** The Hyperkernel enters a **Prioritized Allocation Mode**.
        - It first allocates 100% of the request to `Critical` and `High` priority kernels.
        - It then proportionally allocates the remaining memory among `Normal` and `Idle` kernels, based on their historical usage.
        - `Flagged` kernels (see below) are allocated **0 bytes** – they are effectively frozen until they are executed or recover.
- **The Enforcement Phase:** The Hyperkernel updates the hardware page tables instantly to reflect the new allocations.

### Component D: The Escalation Ladder (The "Three Strikes" Rule)

This is a state machine within the Hyperkernel that defines the lifecycle of a kernel:

| **Stage** | **Trigger Condition (in the tick loop)** | **Hyperkernel Action** |
| :--- | :--- | :--- |
| **Stage 0: Healthy** | Sends heartbeat on time, memory request is reasonable, syscall timestamp is current. | Full memory budget. Normal scheduling priority. |
| **Stage 1: Flagged (Warning)** | Misses **1** heartbeat interval OR memory request is > 50% of total memory. | **Warning:** The Hyperkernel logs the event, marks the kernel as `FLAGGED`, and this tick, it denies the kernel's memory request. It is giving the kernel one tick to prove it's alive. |
| **Stage 2: Deprecated (Freeze)** | Misses **2** consecutive heartbeats **OR** fails to respond to the Stage 1 warning by the next tick. | **Isolation:** The Hyperkernel immediately revokes 100% of the kernel's memory allocation, effectively freezing it. It is placed in a "waiting room" state for one full tick cycle. |
| **Stage 3: Execution (Death)** | Misses a 3rd heartbeat **OR** remains in `DEPRECATED` state for more than one full tick cycle. | **Execution:** The Hyperkernel wipes the kernel's memory region, frees its ID, and releases its CPU handle. The system logs the execution event. A new kernel is automatically spawned for that user if required. |

## 3. Use Cases (Why This Feature is Critical)

| **Scenario** | **How the Hyperkernel Responds** |
| :--- | :--- |
| **A User Kernel Crashes** | The Hyperkernel misses the heartbeat. The kernel goes from `HEALTHY` → `FLAGGED` → `DEPRECATED` → `EXECUTED` in under 1 second. The user is logged out, and 16 MB of RAM is instantly freed and reallocated to other active kernels. |
| **A Kernel Enters a Deadlock** | The kernel's scheduler is stuck, but it might still send a heartbeat. The Hyperkernel's `last_syscall_time` check fails (hung detection). The kernel is immediately `FLAGGED` and `DEPRECATED` in the next tick, breaking the deadlock. |
| **Memory Starvation (Congestion)** | User A requests 80% of system memory. The Hyperkernel grants it. User B then requests 50%. The Hyperkernel detects congestion. It allocates 100% to User A's request and only 20% to User B, based on priority. User B is throttled, but the system remains stable. |
| **Malware/Ransomware Attempt** | Malware tries to allocate hidden memory to encrypt files. The Hyperkernel sees the unapproved memory spike in the next memory request cycle. It `FLAGS` the kernel and denies the request. If the kernel persists, it is `DEPRECATED` and `EXECUTED` before the malware can finish its encryption routine. |

## 4. Edge Cases & Failure Scenarios (What You Must Handle)

| **Edge Case** | **Risk** | **Mitigation Strategy** |
| :--- | :--- | :--- |
| **The "Perfect Storm" (All kernels crash at once)** | The system becomes completely unresponsive. | The Hyperkernel's scheduling loop includes an "Ultimatum Timer." If it detects more than 3 kernels in the `DEPRECATED` state simultaneously, it enters **Emergency Safe Mode**, kills all kernels, and boots a single, minimal "Emergency Recovery" kernel. |
| **False Positive (System Load Jitter)** | A healthy kernel misses a heartbeat due to a severe, temporary CPU spike. | The Hyperkernel's timeout is **not static**. It multiplies the current tick interval by the current `system_load_factor`. If the system is at 90% load, the timeout is extended by 30%. This prevents the system from executing healthy kernels during transient load spikes. |
| **Hyperkernel Overwhelm (DDoS from kernels)** | A malicious kernel sends heartbeats and memory requests at a rate that overwhelms the Hyperkernel's processing logic. | The Hyperkernel implements **Strict Rate Limiting** at the hardware level. It rejects any heartbeat or memory request that arrives faster than 1 per 50ms per kernel ID. The Hyperkernel itself is optimized to process these requests in constant time (O(1)). |
| **Single Kernel Memory Hog** | A kernel requests 100% of available memory legitimately, starving others. | The Hyperkernel's constitution caps individual kernel memory requests at **60% of total system memory**. If a kernel requests more, the request is denied and the kernel is `FLAGGED`. This prevents a single kernel from monopolizing the system. |
| **Kernel Spoofing (Fake Heartbeat)** | A rogue process inside a compromised kernel sends fake heartbeat signals to avoid detection. | The Hyperkernel's heartbeat response packet must be **hardware-anchored**. It must include the kernel's unique cryptographic ID (derived from the CPU's TPM). A packet without this ID is discarded, and the kernel is immediately `FLAGGED` and `DEPRECATED`. |

## 5. Summary: The Hyperkernel as a Living System

This design transforms the Tesseract Hyperkernel from a static hypervisor into a **self-governing ecosystem**. It is a single, immutable, and authoritative entity that monitors, adjusts, and heals itself.

- **It is not a separate process.** It is the process.
- **It is not a watcher.** It is the judge, jury, and executioner.
- **It is not a manager.** It is the sovereign.

This is the architecture that makes Tesseract not just a secure kernel, but a **fundamentally different kind of computing platform**. It is a system that is designed to actively prevent failures, rather than simply react to them.
